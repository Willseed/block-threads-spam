# 目標

封鎖流程要讓使用者快速處置已確認候選，同時避免相似度誤判、帳號切換、重複提交與頁面變更造成誤封。

# 批准前條件

* 使用者應用工作階段有效。
* 使用者已完成近期再驗證。
* Threads 連線有效且登入身分與受保護帳號一致。
* 候選有近期掃描與可讀證據。
* 目標帳號仍存在。
* D1、R2 與 Durable Object 均可用。
* 同一 Threads 連線沒有其他封鎖工作。

# 批准內容

批准必須綁定：

* 應用使用者。
* Threads 連線。
* 受保護正式帳號。
* 候選的完整使用者名稱與平台識別。
* 候選證據版本。
* 批准時間與短期有效期限。
* 只能使用一次的動作識別。

批准不接受萬用字元、相似名稱集合或「封鎖所有高風險候選」等泛用語意。

# 執行階段

1. Durable Object 取得該 Threads 連線的單一封鎖鎖。
2. Workflows 記錄工作已進入不可重複執行的破壞性階段。
3. Browser Run 載入該使用者工作階段並重新確認登入帳號。
4. 系統重新開啟候選個人檔案。
5. 頁面帳號名稱、平台識別與批准不一致時立即停止。
6. 保存封鎖前證據與頁面摘要。
7. 執行一次封鎖操作。
8. 讀取封鎖結果並保存封鎖後證據。
9. 完成 D1 稽核後才將工作標記為成功。

# 人工接管

若 Threads 出現安全驗證或操作介面需要使用者確認，系統暫停並提供只對該工作有效的 Live View。使用者只能在已限制的目標頁面內完成必要操作；接管後系統再次確認目標與結果。

# 結果分類

| 結果 | 意義 | 後續 |
|---|---|---|
| 成功 | 平台介面明確顯示目標已封鎖 | 更新候選、保存證據、通知使用者 |
| 未執行 | 前置條件失敗或動作前停止 | 修復連線或重新掃描後建立新批准 |
| 結果不明 | 可能已點擊，但無法確認最終狀態 | 禁止自動重試，要求使用者在 Threads 或 Live View 中複查 |
| 目標不一致 | 重新載入頁面與批准目標不同 | 安全事件，停止並使該批准失效 |

# 防重複與冪等語意

封鎖是外部平台動作，無法假設安全重試。工作識別、Durable Object 鎖與階段紀錄共同防止雙擊、網路重送或工作重啟產生第二次不確定操作。只有使用者在看過現況後建立新的批准，才可再次執行。
