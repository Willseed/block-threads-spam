# 階段一：登入本服務

使用者以 Email 驗證、Passkey 或其他受支援身分方式登入。首次登入時建立個人租戶，畫面清楚說明本服務需要另一個獨立步驟才能連線 Threads。

成功標準：使用者理解「登入服務」與「登入 Threads」是兩件不同的事。

# 階段二：連線自己的 Threads

使用者閱讀 OAuth token 與資料使用說明，選擇連線 Threads。系統導向 Meta 官方 OAuth 頁面，使用者只在官方頁面登入、完成雙重驗證並決定授權；本服務不顯示帳密欄位。

OAuth callback 完成且官方身分比對成功後，畫面顯示讀取到的 Threads 使用者名稱、顯示名稱與頭像，要求使用者確認這是要保護的正式帳號。若不是，使用者可取消確認、撤銷暫存 token 後重新授權。

成功標準：本服務未要求使用者貼上 Cookie、Session 檔、密碼或驗證碼。

# 階段三：設定監控

系統顯示將採用的有限變形類型與預估候選數量。使用者可啟用或停用部分規則、加入已知疑似帳號、選擇手動掃描或低頻排程，以及設定通知。

成功標準：使用者知道系統會檢查哪些帳號，且候選範圍可見、可控。

# 階段四：檢視候選

儀表板顯示新候選、待審核、高優先候選與工作階段健康狀態。候選卡片顯示完整帳號、差異標示、相似理由、最近檢查時間與證據縮圖。

使用者可：

* 忽略。
* 加入持續監看。
* 開啟完整詳情。
* 進入封鎖確認。

成功標準：使用者不需要在多個 Threads 頁面間手動比對，仍能理解系統為何把帳號列為候選。

# 階段五：批准封鎖

候選詳情頁重新整理個人檔案，顯示正式帳號與目標帳號的完整名稱。使用者完成再驗證，確認只封鎖這一個目標。系統執行後顯示明確成功、失敗或待複查狀態。

若人工封鎖交接能力已經部署驗證並開啟，而 Threads 要求使用者判斷，系統可提供短期 Live View；否則流程失敗關閉。系統不嘗試繞過平台安全控制。

成功標準：使用者在採取動作前看得到目標、理由與最新證據，並能確定不是批次封鎖。

# 階段六：後續管理

活動頁顯示每次掃描、候選決策與封鎖結果。設定頁允許暫停排程、變更候選規則、重新連線或中斷 Threads。工作階段失效時，系統顯示「需要重新登入」，而不是要求提供密碼給客服。

# 體驗上的安全提示

* 所有頁面固定顯示目前受保護 Threads 帳號。
* Live View 頁面提醒使用者只在官方 Threads 網域輸入憑證。
* OAuth 授權頁只接受固定 callback 與一次性 state；返回頁不顯示 code、state 或 token。
* 封鎖按鈕不與忽略、返回等低風險按鈕相鄰。
* 高風險動作需要再驗證與明確完整目標。
* 不使用會誤導使用者以為候選已被 Meta 判定為詐騙的文案。
